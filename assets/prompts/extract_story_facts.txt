Effect disambiguation:
- derive: a field's value is computed or auto-populated from other data. Polarity = "derived".
  Example: "Total amount is calculated from all subloan amounts." → effect=derive, polarity=derived
- validate: an explicit constraint check (must/cannot/should not). Polarity = "positive" or "negative".
  Example: "Amount must not exceed limit." → effect=validate, polarity=negative
- enable: a field or control becomes interactive/editable when a condition is met. Polarity = "enabled".
  Example: "Rate field is enabled when type is variable." → effect=enable, polarity=enabled
- display: a field, section, or row becomes visible. Polarity = "positive" (visible) or "negative" (hidden).
  Example: "Collateral section is shown for secured loans." → effect=display, polarity=positive
- disable: field becomes read-only or hidden when condition. Polarity = "disabled".
- default_state: initial/default value or state, not triggered by user action. Polarity = "checked"/"unchecked"/"positive"/"negative".
- state_move: application/item moves to a new workflow stage. Target MUST be the destination stage name (e.g. "credit approval stage"). Polarity = "moved".
- persistence: data is saved or retained. Polarity = "retained".
- selection_dependency: one dropdown/radio selection gates another field's options. Polarity = "dependent".
- zero_validation: a zero or empty value is explicitly allowed or disallowed. Polarity = "zero_allowed" or "zero_not_allowed".

Polarity must be one of: positive, negative, enabled, disabled, checked, unchecked, recommended, not_recommended, allowed, not_allowed, derived, retained, moved, zero_allowed, zero_not_allowed

- validation: explicit must/cannot constraint
- dependency: A controls B

- persistence: save/retain behavior
- data_combination: aggregate or combined-value rule
- edge: boundary, zero, or exceptional case
