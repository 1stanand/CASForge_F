# CASForge v1 — Accuracy Challenges & Stakeholder Notes

## Target

**v1 goal: ≥ 60% of generated feature files are usable with minor tester review**
(file-level quality metric — the generated file provides a solid draft that the tester
can review and approve without rewriting from scratch).

Individual step grounding (steps copied verbatim from the CAS test repository) is tracked
separately. Steps not found in the corpus get `# [NEW_STEP_NOT_IN_REPO]` markers so testers
know exactly what needs attention. The v2 target is ≥ 90% step-level accuracy.

---

## Why 100% is Not Achievable in v1

### 1. JIRA Story Quality

**Challenge:** CASForge reads what the JIRA story says. If the story is vague, incomplete,
or uses inconsistent terminology, the intent extraction produces weak intents.

**Examples:**
- "The screen should work correctly" → generates generic intents, no specific steps found
- Acceptance criteria written as prose paragraphs with no bullet structure → LLM misses edge cases
- Stories that reference other tickets by number but don't include that context

**Stakeholder impact:** Testers writing good JIRA stories directly improve output quality.
A story with clear, specific acceptance criteria produces 80%+ grounded output.
A vague story may produce 40-50%.

---

### 2. Corpus Coverage Gaps

**Challenge:** CASForge retrieves test steps from the existing CAS ATDD repository.
If a story covers a **new feature** not yet in the repo, the retrieval engine finds
the nearest related scenarios — which may be from a different application stage or context.

**Examples:**
- A new "Recommendation" stage screen that has no existing scenarios at that stage → retrieval
  returns "Credit Approval" stage scenarios (wrong context, more `[NEW_STEP_NOT_IN_REPO]` markers)
- A new UI widget (e.g., dynamic placeholder field) that has no matching steps in the corpus

**Stakeholder impact:** This is expected and correct behaviour. CASForge marks these steps
explicitly so testers know what needs to be written fresh. As more feature files are ingested
into the corpus, coverage improves automatically.

---

### 3. Repository Step Phrasing Variation

**Challenge:** The same logical step is written differently across feature files.
The retrieval engine uses fuzzy matching (vector + FTS + trigram), but subtle phrasing
differences can cause the wrong variant to be selected.

**Examples:**
- `user opens application of "<ProductType>"` vs `user navigates to application screen for "<ProductType>"`
- Different column names in Examples tables (`ProductType` vs `LoanType` vs `LOB`)

**Stakeholder impact:** Low — affects Examples table headers rather than scenario logic.
These are easy to spot and fix in review.

---

### 4. Multi-Stage Stories

**Challenge:** Some stories span multiple application stages (e.g. "applies to DDE and
Credit Approval"). CASForge currently generates one feature file per story. The retrieval
boost is set to the story's primary stage, so steps from the secondary stage get lower scores.

**Stakeholder impact:** Medium. These stories need the tester to review and duplicate
the relevant scenarios per stage. Planned improvement for v2.

---

### 5. LLM Scenario Pruning Quality

**Challenge:** After retrieval selects the best matching scenario from the repo, the LLM
trims it to fit the specific intent. On a weak GPU or with a small LLM context, the LLM
may keep irrelevant steps or drop important ones.

**Examples:**
- Keeps all 12 steps of the source scenario when only 4 are relevant
- Drops the single critical `Then` assertion, triggering the synthesized fallback

**Stakeholder impact:** The synthesized `Then` fallback is marked
`# [SYNTHESIZED_ASSERTION - verify manually]` so testers know to review.
The LLM model quality directly affects this — a larger/better model improves output.

---

### 6. Ordered Flow Complexity

**Challenge:** Ordered (end-to-end journey) flows require scenario steps to chain correctly
across screens. The retrieval engine finds per-screen steps well, but the ordering logic
relies on `order.json` which may not cover all screen transitions for newer features.

**Stakeholder impact:** For v1, unordered flow is more reliable. Use ordered flow only
for well-established journeys already present in the corpus.

---

## What "Grounded" Means

| Marker | Meaning | Action needed |
|--------|---------|---------------|
| *(no marker)* | Step found verbatim in the repo | Ready to use |
| `# [NEW_STEP_NOT_IN_REPO]` | Step not found in corpus | Tester writes/verifies step definition |
| `# [SYNTHESIZED_ASSERTION - verify manually]` | Then clause generated by LLM, not retrieved | Tester validates assertion is correct |

---

## v2 Planned Improvements

1. **Semantic similarity matching for step retrieval** — instead of exact text lookup for
   grounding check, use embedding similarity so "clicks Delete button" matches "removes entry"
2. **Cross-stage scenario generation** — generate separate scenario blocks per stage for
   multi-stage stories
3. **Intent refinement loop** — after first retrieval pass, re-rank intents by corpus coverage
   and flag under-covered areas to the tester before generation
4. **JIRA quality gate** — pre-check JIRA story completeness before extraction and warn
   when acceptance criteria are missing or too vague
5. **Automatic corpus growth** — ingest newly written step definitions immediately after
   tester approves them, so new steps become available for future stories

---

## Summary for Stakeholders

> CASForge v1 produces a working draft .feature file that a tester can review and approve
> in significantly less time than writing from scratch. The v1 quality bar is that **≥ 60%
> of generated files are usable with minor review** — they provide correct structure, real
> repo steps where available, and clear markers on everything that needs tester attention.
> For well-documented stories covering existing functionality, expect **70–85% of steps to
> be grounded** (directly copied from the repo). For new features or corpus gaps, expect
> **40–60% step grounding**, with every ungrounded step explicitly flagged.
> The v2 target is ≥ 90% step-level accuracy through canonical matching, semantic similarity,
> and automatic corpus growth.
